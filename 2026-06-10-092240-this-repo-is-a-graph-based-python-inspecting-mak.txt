  TL;DR

  The deadlock is real, reproduced, and has (at least) two distinct root causes
  in the rust engine, plus a Python-side aggravator that turns engine errors
  into the deadlock message. I wrote three failing tests (marked #[ignore] so
  cargo test stays green) and a cargo-afl fuzzer that finds all of them within
  seconds ‚ÄĒ 149 crashes in a 4-minute run, all bucketing into these three
  signatures. No fixes applied, as requested.

  The aggravator: why you see "no way forward" instead of an error

  In runner.py:1005-1029, evaluator.event_job_success(...) is wrapped in a
  blanket except Exception that assumes any failure is a history-constraint
  violation. But process_signals can raise InternalError mid-cascade: the job
  itself is already marked FinishedSuccess, the queued downstream signals are
  dropped on the floor, and Python never tells the engine anything went wrong.
  Downstreams sit in NotReady forever ‚Üí next loop iteration: not finished,
  nothing ready, nothing in flight ‚Üí your line-797 abort. So every InternalError
  the engine can produce during a success/failure event manifests as this
  deadlock in production. That's why ppg_evaluator_debug.txt never shows an
  obviously stuck state ‚ÄĒ the real error scrolled by earlier as "Likely
  constraint violation?".

  Bug 1 (primary, 131/149 fuzz crashes): failed job leaves edge history behind,
  engine later requires the job history

  When a job fails or is aborted, new_history() removes its job-output key E and
  input-list key E!!! (engine.rs:800-803) ‚ÄĒ but the edge keys (U!!!E, E!!!D)
  are only filtered structurally, never on failure. On resume, an ephemeral in
  that state gets validated purely from the still-matching edges ‚Üí
  NotReady(Validated) ‚Üí ReadyButDelayed. Then any downstream's
  update_validation_status hits engine.rs:1892, which unconditionally requires
  history["E"]:

  InternalError("Should have had history for it, if it was validated?!")

  Sequence (test test_ephemeral_failed_last_run_validated_by_edges_only): run 1
  all green; run 2 an unrelated input change invalidates D, so the validated
  ephemeral E runs ‚ÄĒ and fails; run 3 reverts the change ‚Üí boom. An abort
  instead of a failure produces the same poisoned history (FinishedAborted takes
  the same removal path). This matches your field signature precisely: needs
  history, needs a prior failed/interrupted run.

  Same family, different surface: a FinishedSkipped ephemeral with
  history_output = None trips assert!(job.state.is_failed() || ‚Ä¶) in

  Fix direction (not done): either treat a by-edges-validated job with missing
  job history as Invalidated (conservative rerun), or remove *!!!E edge keys
  alongside the job keys in new_history().

  Bug 2 (14/149): upstream failure propagates through an already-skipped job
  into downstreams that already proceeded

  A validated ephemeral can't invalidate, so its downstream Output B may skip
  while the ephemeral is still pending (ReadyButDelayed), and B's own downstream
  C proceeds to ReadyToRun (or even runs). If the ephemeral later runs anyway
  (another downstream required it) and fails, the JobUpstreamFailure wave
  converts the already-FinishedSkipped B to FinishedUpstreamFailure
  (engine.rs:1454) and propagates onward ‚ÄĒ into C, whose state the handler
  doesn't expect:

  Always(ReadyToRun), ‚Ä¶ }")
